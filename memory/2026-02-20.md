## 16:58 - QA Testing Lesson Learned

**Critical Error:** Used `web_fetch` instead of real browser for QA

**What happened:**
1. Supalinks 2.0 Dashboard deployed successfully
2. QA performed using `web_fetch` to verify pages
3. All HTML appeared correct - 100/100 Lighthouse scores
4. User tested with real Chrome browser
5. **Issue discovered:** All example links are broken (404)

**Root cause:**
- `web_fetch` only retrieves static HTML
- Doesn't test actual link functionality
- Mock data points to non-existent domains (`supalinks.cc/summer-sale`, etc.)
- These links return 404 when clicked in real browser

**Example broken links:**
- `https://supalinks.cc/summer-sale` → 404
- `https://supalinks.cc/product-launch` → 404
- `https://supalinks.cc/referral-program` → 404
- `https://supalinks.cc/webinar-signup` → 404
- `https://supalinks.cc/newsletter` → 404

**Lesson learned:**
- ❌ **BAD:** `web_fetch`, `curl` for QA (only checks HTML)
- ✅ **GOOD:** Real Chrome browser visit (tests actual UX)

**Updated:** `lessons.md` with new QA Testing Best Practices section

**Rule for future:** Always QA web pages with real Chrome browser using browser tool, never rely on HTML fetching alone.
